# Anwenderhandbuch m+m 8D Reporting

## Einleitung
Das Modul unterstützt Anwender bei der Erfassung, Steuerung und Nachverfolgung von Reklamationen, Rückläufern und 8D-Dokumenten in Microsoft Dynamics 365 Business Central. Im Mittelpunkt stehen Reklamationsprofile, die Zuordnung zu Debitoren und Belegen, die Dokumentation der 8D-Inhalte sowie die Ausgabe von Bericht und E-Mail.

Die tägliche Arbeit beginnt meist an einer Rücksende- oder Reklamationssituation. Danach werden Profile, Teams, Ursachen, Korrekturmaßnahmen und Texte gepflegt, bevor der Anwender Zeilen prüft, Berichte erzeugt oder E-Mails versendet. Die Einrichtung legt fest, welche Nummernserien, Sperrlogik, Berichtsvorlagen und Mailbausteine im Prozess verwendet werden.

## Schnellstart für Anwender
1. Öffnen Sie zuerst die **Rekl.-Verw. Einrichtung** und prüfen Sie die zentrale Aktivierung, die Nummernserien, die Sperrlogik und die Berichtsvorgaben.
2. Pflegen Sie die Stammdaten für **Reklamationsgruppen**, **Reklamationsdisziplinen**, **Korrekturmaßnahmen**, **Reklamation Fehler**, **Reklamationsteam** und **Reklamationstext**.
3. Legen Sie ein **Reklamationsprofil** an oder öffnen Sie ein vorhandenes Profil und pflegen Sie die Linien unter **Reklamationsprofil**.
4. Weisen Sie das Profil über die Zuordnungsliste den betroffenen Belegen oder Debitoren zu.
5. Öffnen Sie die betroffene Verkaufszeile in **Verkaufszeile Ablehnungsantrag**, prüfen Sie die Daten und ändern Sie bei Bedarf den Status.
6. Erzeugen Sie anschließend den **8D-Reklamationsbeleg** oder die **Reklamationsmail** und dokumentieren Sie den Versand in der **8D Reklamationsbelegposten**-Liste.

## Einrichtung
Die Einrichtung wird von einem Administrator oder Key User vor der ersten produktiven Nutzung gepflegt. Sie legt fest, welche Berichte, Nummernserien, Sperrregeln und Mailvorlagen der Reklamationsprozess verwendet. Die Seite **Rekl.-Verw. Einrichtung** ist eine globale Einrichtungsseite und wird direkt geöffnet.

### Reklamationsprofile einrichten
Diese Einrichtung pflegt der Key User, bevor Profile im Tagesgeschäft verwendet werden. Das Profil bestimmt, wie viele Disziplinen ein 8D-Profil haben darf, welche Laufzeit für Berichte gilt und wann ein Profil bearbeitet oder abgeschlossen werden darf. Die Folge ist die Pflege von Profilen, die später an Debitoren oder Belege gebunden werden.

1. Legen Sie in **Reklamationsprofile** ein neues Profil an. Beim Anlegen vergibt das System die Nummer aus der Nummernserie und setzt den Status auf **Neu**.
2. Tragen Sie **Beschreibung**, **Art** und **Berichtsintervall** ein. Wenn Sie die Art wählen, bestimmt sie die maximale Anzahl der Disziplinen im Profil.
3. Pflegen Sie die Zeilen in der Unterseite **Reklamationsprofil**. Jede Zeile übernimmt eine Disziplin und ergänzt eine Bemerkung für den Reklamationsablauf.
4. Ändern Sie den Status erst, wenn die Zeilenanzahl zur gewählten Art passt. Solange der Status **Neu** oder **In Bearbeitung** ist, dürfen Sie Profil und Zeilen noch anpassen.

| Feld | Erläuterung |
|---|---|
| Nr. | Wenn das System eine Nummer vergibt, ist das Profil eindeutig identifizierbar und kann in Zuordnungen und Folgebelegen verwendet werden. |
| Beschreibung | Wenn Sie eine Beschreibung pflegen, erkennen Anwender das Profil in Listen und Auswahldialogen schneller und vermeiden Verwechslungen. |
| Art | **1D:** ein Profil mit einer Disziplin; **2D:** zwei Disziplinen; **3D:** drei Disziplinen; **4D:** vier Disziplinen; **5D:** fünf Disziplinen; **6D:** sechs Disziplinen; **7D:** sieben Disziplinen; **8D:** acht Disziplinen. Die gewählte Art begrenzt die Anzahl der Zeilen, die Sie im Profil pflegen dürfen. |
| Status | **Neu:** Das Profil kann angelegt und gepflegt werden. **Zertifiziert:** Das Profil ist freigegeben und darf nicht mehr geändert werden. **In Bearbeitung:** Das Profil ist noch offen für Anpassungen. **Geschlossen:** Das Profil ist abgeschlossen und dient nur noch als Referenz. |
| Anz. Zugew. Debitoren | Wenn dieser Wert steigt, ist das Profil bereits mehreren Debitoren zugeordnet und wird in der Anwendung entsprechend häufiger vorgeschlagen. |
| Berichtsintervall | Wenn Sie hier eine Frist pflegen, steuert sie den Abstand zwischen den Berichtsläufen und den nächsten Fälligkeitszeitpunkt. |

#### Reklamationsprofile kopieren
Die Aktion **Profil kopieren** nutzen Sie, wenn Sie ein bestehendes Profil als Vorlage für ein neues Profil oder für ein Debitorenprofil verwenden möchten.

1. Öffnen Sie das gewünschte **Reklamationsprofil**.
2. Wählen Sie **Profil kopieren**.
3. Entscheiden Sie sich für **Kopie in neues Profil** oder **Kopie in neues Debitorenprofil**.
4. Das System legt die Kopie an, übernimmt die Zeilen und öffnet anschließend die neue Karte zur Nachbearbeitung.

2. Pflegen Sie **Beschreibung**, **Art** und **Berichtsintervall**.
3. Erfassen Sie die Zeilen in der Unterseite **Reklamationsbelegprofile**.
4. Wählen Sie den Status erst dann um, wenn die Zeilen vollständig sind.

| Feld | Erläuterung |

| Beschreibung | Wenn Sie die Beschreibung pflegen, erkennt der Anwender den Zweck des Rücksende-Auftrags-Profils sofort in der Auswahlliste. |
| Art | **1D:** eine Disziplin wird abgedeckt. **2D:** zwei Disziplinen sind zulässig. **3D:** drei Disziplinen sind zulässig. **4D:** vier Disziplinen sind zulässig. **5D:** fünf Disziplinen sind zulässig. **6D:** sechs Disziplinen sind zulässig. **7D:** sieben Disziplinen sind zulässig. **8D:** acht Disziplinen sind zulässig. Die Auswahl begrenzt die spätere Zeilenanzahl. |
| Status | **Neu:** Das Profil ist noch frei bearbeitbar. **Zertifiziert:** Das Profil ist für den operativen Einsatz fixiert. **In Bearbeitung:** Das Profil kann weiter gepflegt werden. **Geschlossen:** Das Profil dient nur noch zur Nachverfolgung. |
| Anz. Zugew. Debitoren | Wenn dieser Wert gefüllt ist, zeigt er die Zahl der Debitoren, die bereits dieses Rücksendeprofil verwenden. |
| Berichtsintervall | Wenn Sie hier ein Intervall setzen, übernimmt der Versand- oder Berichtslauf den zeitlichen Abstand aus diesem Profil. |

| Ursprünglicher Disziplintext | Wenn dieser Text angezeigt wird, erkennt der Anwender die Quelle der übernommenen Disziplin und kann die Änderung im Profil nachvollziehen. |

| Disziplin beendet | Wenn Sie das Kennzeichen setzen, kann die Zeile als abgeschlossen behandelt werden; ohne Kennzeichen bleibt sie offen. |
| Disziplin beendet am | Wenn dieses Datum gefüllt ist, ist der Abschluss zeitlich dokumentiert und die Zeile kann in Berichten als erledigt erscheinen. |

## Stammdaten
Die Stammdaten werden vor der täglichen Nutzung gepflegt und steuern die Auswahl im Hauptbeleg, die Teamzuordnung und die Textverarbeitung. Sie sind nicht nur Nachschlagewerke, sondern bestimmen den späteren Ablauf der Reklamationsbearbeitung.

### Reklamationsgruppen
Reklamationsgruppen dienen als fachliche Klammer für ähnliche Fälle. Der Anwender pflegt sie, bevor Profile oder Belege auf eine Gruppe zurückgreifen. Über die Gruppe wird im Prozess sichtbar, welche Klassifizierung für die Auswertung gilt.

### Reklamationsdisziplinen
Reklamationsdisziplinen definieren, welche fachlichen Schritte in einem Profil bearbeitet werden. Die Beschreibung wird in Profileinträgen übernommen und trägt die fachliche Aussage bis in die tägliche Arbeit.

### Korrekturmaßnahmen
Korrekturmaßnahmen bilden die Maßnahmen ab, die aus einer Reklamation folgen. Sie werden in der Dokumentation verwendet und unterstützen die spätere Nachverfolgung.

### Reklamation Fehler
Reklamation Fehler steuern, welche Fehlerursachen ein Anwender im Prozess auswählen kann. Die Auswahl beeinflusst die spätere Auswertung und die Ableitung von Maßnahmen.

### Reklamationsteam und Mitglieder
Die Teamstammdaten legen fest, wer im 8D-Prozess verantwortlich ist und wer Freigaben erteilen darf. Das ist wichtig, weil die Codeunit **MuM 8D Reporting** bei bestimmten Statuswechseln prüft, ob ein Anwender Teamleiter oder autorisiert ist.

| Feld | Erläuterung |
|---|---|
| Teamcode | Wenn Sie einen Teamcode pflegen, wird die Gruppe der verantwortlichen Personen eindeutig identifiziert. |
| Teammitglied | Wenn Sie ein Mitglied zuordnen, kann dieses im Ablauf als zuständige Person oder Freigabeberechtigte verwendet werden. |
| Teamleiter | Wenn das Kennzeichen gesetzt ist, darf das Mitglied in den geprüften Statusfällen den Vorgang freigeben oder ablehnen. |
| Freigabeautorisation | Wenn das Kennzeichen gesetzt ist, kann das Mitglied eine Freigabe für den Status **Genehmigt** erteilen. |
| Genehmigungsautorisierung | Wenn das Kennzeichen gesetzt ist, kann das Mitglied Reklamationen im Genehmigungsweg weiterführen. |
| Ablehnungsautorisierung | Wenn das Kennzeichen gesetzt ist, kann das Mitglied Reklamationen im Ablehnungsweg verarbeiten. |

### Reklamationstext
Reklamationstexte dienen als Bausteine für spätere Druck- und Mailausgaben. Sie werden vor dem Versand gepflegt, damit die Dokumente und E-Mails die fachlich richtigen Formulierungen enthalten.

### Ursache, Gruppe und Zuordnung
Die Seiten **Reklamationsgruppe**, **Reklamationsdisziplin**, **Reklamation Fehler**, **Korrekturmaßnahmen** und die Zuordnungstabellen unterstützen die fachliche Einordnung einer Reklamation. Der Anwender pflegt diese Stammdaten zuerst, damit die Profile später ohne Medienbruch in der täglichen Arbeit genutzt werden können.

## Profile und Zuordnungen
Profile steuern, wie Reklamationen im Prozess verwendet werden. Die Zuordnungsliste verbindet das Profil mit dem Debitor oder dem Beleg und ist damit der Übergang von der Stammdatenpflege in den operativen Ablauf.

### Debitorenprofile zuordnen
Pflegen Sie in **Debitorenreklamationsprofile** die Zuordnung zwischen Debitor und Reklamationsprofil. Diese Pflege erfolgt durch den Key User oder Administrator, bevor Anwender in Belegen mit der Reklamationslogik arbeiten.

### Profilermittlung
Die Codeunit prüft bei der Freigabe, welche Profile an den betroffenen Verkaufszeilen hängen und welche Teams für den aktuellen Status zuständig sind. Dadurch wird die Prozesslogik über die Profilzuordnung aufgelöst und nicht manuell im Beleg improvisiert.

## Tägliche Arbeit
Die tägliche Arbeit beginnt an der betroffenen Verkaufszeile oder am Rücksendeauftrag. Der Anwender prüft dort Status, Belegbezug, Disziplinen und Kosten und entscheidet anschließend, ob die Zeile in den Reklamationsfluss übernommen oder weiter bearbeitet wird.

### Verkaufszeile Ablehnungsantrag
Öffnen Sie **Verkaufszeile Ablehnungsantrag**, wenn Sie die vom System gefilterten Rücksende- und Ablehnungszeilen bearbeiten möchten. Die Seite zeigt nur Verkaufszeilen mit dem Status **Ablehnungsantrag** und dient damit als Einstieg in die operative Prüfung.

| Feld | Erläuterung |
|---|---|
| Reklamationsstatus | **Offen:** Die Zeile ist noch nicht gestartet und wartet auf Bearbeitung. **In Bearbeitung:** Die Zeile wird aktiv geprüft und kann im nächsten Schritt geändert werden. **Abgelehnt:** Die Zeile ist fachlich verworfen und wird nicht weiter freigegeben. **Ablehnungsantrag:** Die Zeile wurde für die Ablehnung vorgemerkt und wartet auf die Statusänderung. **Genehmigt:** Die Zeile ist freigegeben und kann in nachgelagerte Prozesse laufen. **Gutschrift genehmigt:** Der Vorgang ist für eine Gutschrift freigegeben. **Gutschriftantrag:** Der Vorgang wartet auf die Gutschriftfreigabe. |
| Belegart | Wenn Sie hier die Belegart sehen, erkennen Sie, ob die Zeile zu einem Rücksendeauftrag gehört und wie sie im weiteren Ablauf behandelt wird. |
| Verk.-an Debitor-Nr. | Wenn diese Nummer angezeigt wird, ist die betroffene Debitorenposition eindeutig dem Rücksendevorgang zugeordnet. |
| Belegnr. | Wenn die Belegnummer gefüllt ist, kann der Anwender die Zeile direkt im Ursprungsvorgang nachverfolgen. |
| Zeilennr. | Wenn die Zeilennummer sichtbar ist, bleibt die einzelne Verkaufszeile im Prozess eindeutig referenzierbar. |
| Typ | Wenn Sie den Typ sehen, erkennen Sie, ob es sich um einen Artikel, eine Position oder eine andere Zeilenart handelt. |
| Nr. | Wenn die Nummer angezeigt wird, ist der zugrunde liegende Artikel oder Beleginhalt eindeutig. |
| Lagerortcode | Wenn der Lagerortcode gefüllt ist, ist der physische Bezug für die Rücksende- oder Reklamationsbearbeitung klar. |
| Buchungsgruppe | Wenn die Buchungsgruppe gefüllt ist, wird die finanzielle Zuordnung in der Buchung eindeutig bestimmt. |
| Versanddatum | Wenn das Versanddatum gesetzt ist, erkennt der Anwender den zeitlichen Bezug für die Reklamationsbewertung. |
| Beschreibung | Wenn die Beschreibung angezeigt wird, sieht der Anwender den fachlichen Inhalt der Zeile ohne Rücksprung in den Ursprungsbeleg. |
| Beschreibung 2 | Wenn ein zweiter Beschreibungstext vorhanden ist, ergänzt er die fachliche Einordnung der Zeile. |
| Einheit | Wenn die Einheit angezeigt wird, erkennt der Anwender die Mengeneinheit der Zeile und kann die Menge korrekt bewerten. |
| Menge | Wenn die Menge angezeigt wird, sieht der Anwender die betroffene Menge und kann die Rückmeldung in der passenden Größenordnung verarbeiten. |
| Einzelpreis | Wenn der Einzelpreis gefüllt ist, erkennt der Anwender den wertmäßigen Umfang der Zeile. |
| Betrag | Wenn der Betrag angezeigt wird, sieht der Anwender den Netto-Wert der Verkaufszeile. |
| Betrag inkl. MwSt. | Wenn der Bruttobetrag angezeigt wird, erkennt der Anwender den Gesamtwert einschließlich Steuer. |
| Kz. Dimension 1 | Wenn die erste Dimensionskennzeichnung gesetzt ist, wird die Zeile in der Kosten- oder Standortauswertung richtig eingeordnet. |
| Kz. Dimension 2 | Wenn die zweite Dimensionskennzeichnung gesetzt ist, wird die Zeile für Auswertungen und Folgebelege weiter eingegrenzt. |

#### Status ändern
Mit **Status ändern** übernimmt das System die Reklamationsstatuslogik für die markierte Zeile. Die Aktion wird genutzt, wenn die Zeile von der Anforderung in die eigentliche Bearbeitung wechseln soll.

1. Markieren Sie die Verkaufszeile.
2. Wählen Sie **Status ändern**.
3. Das System prüft die Zuständigkeit anhand des aktuellen Status und der Teamzuordnung.
4. Nach erfolgreicher Prüfung wird der Status im Beleg aktualisiert.

### Rücksendeauftragsprofile in der täglichen Arbeit
Wenn die Verkaufszeile bereits einem Profil zugeordnet ist, greift die Status- und Teamlogik aus den Profilstammdaten. Der Anwender muss dann nicht manuell nachregeln, sondern arbeitet im vorgesehenen 8D-Ablauf.

## 8D-Report und E-Mail
Der Bericht und der Mailversand bilden den Abschluss vieler Reklamationsvorgänge. Das System protokolliert dabei Druckzeit, Benutzer und Intervall, damit spätere Nachweise verfügbar sind.

### 8D-Reklamationsbeleg
Der 8D-Bericht wird auf Basis der in der Einrichtung hinterlegten Berichtsnr. erzeugt. Der Anwender nutzt ihn, wenn die Reklamation als Dokument ausgegeben werden soll.

| Feld | Erläuterung |
|---|---|
| Eintrag-Nr. | Wenn ein Eintrag erzeugt wird, ist der Druckvorgang in der Protokolltabelle eindeutig nachvollziehbar. |
| Verkaufsart | **Einkauf:** Der Eintrag bezieht sich auf einen Einkaufsbeleg. **Verkauf:** Der Eintrag bezieht sich auf einen Verkaufsbeleg. Die Auswahl steuert, aus welcher Prozessrichtung der Bericht stammt. |
| Belegart | Wenn die Belegart gefüllt ist, ist der verknüpfte Verkaufs- oder Einkaufsbeleg im Log eindeutig erkennbar. |
| Belegnr. | Wenn die Belegnummer gefüllt ist, lässt sich der gedruckte Vorgang auf den Ursprungsbeleg zurückführen. |
| Zeilennr. | Wenn die Zeilennummer gefüllt ist, wird der einzelne Posten im Bericht eindeutig referenzierbar. |
| Berichtsnr. | Wenn die Berichtsnr. gespeichert wird, ist klar, welche Ausgabe tatsächlich erzeugt wurde. |
| Druckdatum | Wenn das Druckdatum gespeichert wird, sieht der Anwender, wann der Bericht erzeugt wurde. |
| Druckuhrzeit | Wenn die Druckuhrzeit gespeichert wird, ist der Erzeugungszeitpunkt sekundengenau dokumentiert. |
| Gedruckt von | Wenn der Benutzer gespeichert wird, ist nachvollziehbar, wer den Bericht gestartet hat. |
| Berichtsintervall | Wenn das Intervall gespeichert wird, ist klar, in welchem Rhythmus die nächste Ausgabe fällig ist. |

### Reklamationsmail
Der E-Mail-Versand verwendet die in der Einrichtung hinterlegten Namensbausteine und den Standardbericht. Das System erstellt die PDF-Datei, ermittelt die sprachabhängigen Bezeichnungen und versendet die Mail an die Debitorenadresse.

### Versandprotokollierung
Der Versand wird in der Logtabelle und in den Datumsfeldern der Verkaufszeile nachvollziehbar. So kann der Anwender erkennen, wann der Bericht zuletzt versendet wurde und wann der nächste Versand ansteht.

## Lager, Sperrbestand und Folgebelege
Die Lager- und Folgebeleglogik wird eingesetzt, wenn Reklamationskosten, Transport oder Sperrbestand in nachgelagerte Belege übergehen. Die Einrichtung bestimmt dabei, ob Lagerorte, Lagerplätze und Buchungsblätter gesperrt werden.

### Transportauftrag erzeugen
Wenn eine Verkaufsretoure einen Transport benötigt, legt das System auf Basis der in der Einrichtung hinterlegten Sachkontonr. einen Einkaufsauftrag an. Der Anwender muss dafür den Transport nur dann starten, wenn der Debitor und die Transportroute korrekt gepflegt sind.

1. Prüfen Sie in der Verkaufsretoure die **Debitorennr. für Reklamationen**.
2. Stellen Sie sicher, dass in **Rekl.-Verw. Einrichtung** ein Sachkonto für den Transport hinterlegt ist.
3. Wählen Sie die Aktion zum Erzeugen des Transportauftrags.
4. Das System legt den Einkaufsauftrag an, übernimmt die Lieferadresse und öffnet den Beleg.

### Lagerbestand sperren
Wenn **Lagerbestand sperren** aktiv ist, verhindert die Einrichtung weitere freie Lagerbewegungen an den betroffenen Orten oder Lagerplätzen. Das schützt den Reklamationsbestand vor unbeabsichtigten Folgebewegungen.

### Umlagerungsjournal
Die Umlagerungsbuchung wird über die hinterlegte Buch.-Blattvorlage und den Buch.-Blattstapel verarbeitet. Dadurch wird der gesperrte Bestand nicht in einem beliebigen Journal, sondern im vorgesehenen Reklamationsjournal geführt.

## Historie, Archiv und Nachverfolgung
Die Nachverfolgung erfolgt über Protokolle, Archivseiten und Historienlisten. So kann der Anwender später nachvollziehen, wann ein Bericht gedruckt, eine Zeile abgeschlossen oder ein Vorgang archiviert wurde.

### 8D Reklamationsbelegposten
Die Liste **8D Reklamationsbelegposten** enthält die Druck- und Versandspuren des 8D-Berichts. Der Anwender erkennt dort Belegart, Belegnummer, Benutzer und Berichtsintervall.

### Reklamationshistorie
Die Historie hilft bei der Rückverfolgung vergangener Vorgänge. Sie ergänzt die aktuelle Bearbeitung um archivierte Informationen und unterstützt die Suche nach abgeschlossenen Fällen.

### Archive
Die Archivseiten für Verkaufsbelege und Rücksendeprozesse sichern die frühere Belegstruktur. Der Anwender nutzt sie, wenn er einen abgeschlossenen Vorgang mit unveränderten Daten erneut prüfen muss.

## FAQ und Troubleshooting
### Warum kann ich ein Reklamationsprofil nicht mehr ändern?
Ein Profil darf nur geändert werden, solange der Status **Neu** oder **In Bearbeitung** ist. Sobald es **Zertifiziert** oder **Geschlossen** ist, sperrt das System Änderungen, damit der operative Prozess stabil bleibt.

### Warum lässt sich ein Profil nicht löschen?
Wenn bereits eine Profilzuordnung existiert, verhindert das System das Löschen. Dadurch bleiben Belegbeziehungen und Auswertungen konsistent.

### Warum kommt beim Statuswechsel eine Berechtigungsprüfung?
Die Codeunit prüft bei den Statusen **Genehmigt**, **Gutschrift genehmigt** und **Abgelehnt**, ob der aktuelle Benutzer Teamleiter oder fachlich autorisiert ist. Ohne passende Teamrolle wird der Wechsel nicht zugelassen.

### Warum wird beim Anlegen eines Profils automatisch eine Nummer vergeben?
Die Nummer stammt aus der in der Einrichtung hinterlegten Nummernserie. So bleibt jedes Profil eindeutig und kann später in Zuordnungen und Berichten referenziert werden.

### Warum kann ich keine neue Disziplinzeile anlegen?
Die Anzahl der Zeilen ist durch die gewählte Profilart begrenzt. Wenn die maximale Disziplinenzahl erreicht ist, blockiert das System weitere Zeilen.

### Warum springt der Status nicht auf abgeschlossen?
Der Abschluss ist nur möglich, wenn die Zeilenanzahl zur gewählten Profilart passt. Andernfalls verhindert das System den Statuswechsel, damit das Profil fachlich vollständig bleibt.

### Warum erscheint der Transportauftrag nicht?
Der Transportauftrag kann nur entstehen, wenn in der Einrichtung ein Transport-Sachkonto gepflegt ist und die Verkaufsretoure eine Debitorennr. für die Reklamation enthält. Fehlt eine dieser Angaben, bricht der Ablauf ab.

### Warum wird der Bericht nicht wie erwartet erzeugt?
Der Bericht wird über die in der Einrichtung hinterlegte Berichtsnr. gestartet. Wenn dort kein gültiger Bericht gepflegt ist, kann der 8D-Druck nicht korrekt ausgeführt werden.

### Warum wird im E-Mail-Versand kein passender Dateiname verwendet?
Der Dateiname wird aus den in der Einrichtung gepflegten Mail-Namensbausteinen und der Sprache des Debitors zusammengesetzt. Fehlt die Zuordnung, verwendet das System keinen sprachspezifischen Zielnamen.

### Warum bleibt eine Verkaufszeile in der Liste sichtbar?
Die Liste **Verkaufszeile Ablehnungsantrag** zeigt nur Zeilen mit dem passenden Reklamationsstatus. Sobald der Status geändert oder die Zeile aus dem Filterbereich fällt, verschwindet sie aus der Liste.

## Typische Anwendungsfälle
### Reklamation aus einem Telefonanruf
Ein Anwender erhält einen Anruf von einem Debitor zu einer beanstandeten Lieferung. Er öffnet die betroffene Verkaufszeile in **Verkaufszeile Ablehnungsantrag**, prüft den Status und ordnet bei Bedarf ein Reklamationsprofil zu. Danach ändert er den Status und erzeugt den Bericht für die interne Nachverfolgung.

### Reklamation aus einer E-Mail
Eine eingehende E-Mail beschreibt einen Qualitätsmangel. Der Key User öffnet das passende **Reklamationsprofil**, kontrolliert die Disziplinen und nutzt anschließend **Profil kopieren**, um einen neuen Fall schneller anzulegen. Danach wird die Zeile im Tagesgeschäft weiterbearbeitet und bei Bedarf als 8D-Bericht ausgegeben.

### Rücksendevorgang mit Transport
Der Anwender erhält einen Rücksendeauftrag, der zusätzliche Transportkosten auslösen soll. Er prüft zuerst die **Rekl.-Verw. Einrichtung**, anschließend die Debitorenzuordnung und erzeugt dann den Transportauftrag. Das System übernimmt die Adressdaten, öffnet den Folgebeleg und dokumentiert die Bewegung im Prozess.

### Abschluss mit Bericht und E-Mail
Nach der fachlichen Freigabe erzeugt der Anwender den **8D-Reklamationsbeleg** und versendet parallel die **Reklamationsmail**. Der Versand wird protokolliert, und der Anwender kann später in den Log- und Archivlisten nachvollziehen, wann der Bericht erstellt und verschickt wurde.
